The Project — PaceCircle

What You're Building

A Fitness Calendar with Low-Pressure Social Accountability

PaceCircle is a web app where you plan workouts on a personal calendar and belong to small, invite-only circles — your running group, your Pilates friends, your work gym crew. You choose which workouts to share with each circle. Circle members see shared events in a Feed tab and can tap "Count me in" to RSVP. Both the event creator and all joiners see who's attending. RSVPs auto-expire when the event time passes. No algorithm, no public search, no follower counts — just your schedule and your people.

📅
Personal Calendar — your workouts only, like Google Calendar. Click any day to log an event or see existing ones. Activity types: RUN, WALK, CYCLING, PILATES_BARRE, STRENGTH_GYM, YOGA, SWIM, OTHER.
🔵
Invite-Only Circles — create a circle, get a join code. Friends join via link or code only. No public search, no discovery. One workout can be posted to multiple circles simultaneously.
📣
Circle Feed — a separate tab showing upcoming shared workouts from all your circles. Creator, activity type, time, optional meetup location. One-tap RSVP.
✋
"Count me in" RSVP — both the event creator and all joiners see each other's names on the attendee list. Mutual visibility. Low stakes. Auto-expires when the event time passes.
⏱️
Background Jobs — nightly scheduled job cleans expired RSVP records. Weekly job deactivates dormant circles. Background async email sends so HTTP responses are never blocked.
🔒
Privacy by Design — JWT-authenticated API, strict per-user data isolation. A valid token from User A cannot access User B's private workouts. Leaving a circle removes feed visibility immediately.

Week 1 Project Setup + Layered Architecture ~10 hrs
Mon / Tue
Read: Spring Boot project structure — what every folder and file does. If you have an existing Spring project, annotate every class and annotation: what does this do? Don't look it up yet — guess, then verify.
Wed / Thu
Build: Create the PaceCircle project at start.spring.io (Maven, Java 21, Spring Web + Spring Data JPA + PostgreSQL Driver + Validation). Define your entities: User, WorkoutEvent (with activityType enum: RUN / WALK / CYCLING / PILATES_BARRE / STRENGTH_GYM / YOGA / SWIM / OTHER, plus optional customActivityLabel), Circle, CircleMembership, WorkoutCircle (event↔circle join table), WorkoutJoin ("Count me in" RSVP). Write entity classes, repositories, and service stubs — no real logic yet.
Fri
Understand: What is @Transactional and when does it matter? Write a test that demonstrates a transaction rolling back when an exception is thrown. This appears in almost every Java backend project.
Weekend
Build + Verify: GET /v1/workout-events and POST /v1/workout-events. Verify both in Postman. Read Baeldung's Spring Boot project structure guide and the Spring Data JPA docs on repository methods.
@Transactional
Wraps a method in a database transaction — if anything throws, the whole operation rolls back. Use it on service methods that write to the DB. Project gotcha: calling a @Transactional method from within the same class bypasses Spring's proxy — the transaction won't fire. This is one of the most common Spring project questions.
Controller → Service → Repository
The three-layer pattern every Spring Boot app uses. Controller handles HTTP. Service holds business logic. Repository talks to the DB. Never put SQL or business rules in a controller. Never call a repository from a controller.
@Entity + JPA Relationships
@OneToMany, @ManyToOne — User has many WorkoutEvents. Understand FetchType.LAZY vs EAGER. LAZY (default) loads related data only when accessed. EAGER loads everything upfront, which triggers the N+1 query problem in poorly written code.
🏁 Week 1 Milestone
PaceCircle project running locally. All 6 entities in PostgreSQL. GET and POST for WorkoutEvents working in Postman. You can explain the @Transactional internal-call gotcha.
Week 2 java.time, Streams, and the Standard Library ~10 hrs
Mon / Tue
Learn java.time in depth. The old Date and Calendar classes are legacy — every real codebase uses java.time. Drill: Instant, ZonedDateTime, LocalDate, Duration, Period. Write examples specific to PaceCircle: "get all events in the next 7 days," "check if an RSVP is expired," "calculate days since the last workout."
Wed / Thu
Java Streams. Write service methods using filter(), map(), collect(), groupingBy(), sorted(). Example: "given a list of upcoming events, group them by activityType and count each type." No raw for-loops where Streams work cleanly.
Fri + Weekend
Build into PaceCircle: GET /v1/workout-events?from=2025-04-01&to=2025-04-30 (calendar date range), GET /v1/workout-events/upcoming (next 7 days), and RSVP expiry filtering — exclude WorkoutJoin records where the event's scheduledAt is in the past. All using java.time and Streams.
Instant vs ZonedDateTime
Instant is a UTC timestamp (use for when a record was created). ZonedDateTime is a date + time + timezone (use for when a workout is scheduled). Store scheduledAt as ZonedDateTime — a 6pm run in Charlotte is a different moment than 6pm in Chicago. Getting this wrong breaks expiry logic across timezones.
Stream.collect(groupingBy)
Groups a list into a Map by a key. events.stream().collect(Collectors.groupingBy(WorkoutEvent::getActivityType, Collectors.counting())) — event count per activity type in one line. This pattern appears constantly in analytics-style endpoints.
🏁 Week 2 Milestone
Date-range filtering works on all workout event endpoints. Upcoming events endpoint returns the next 7 days. Expired RSVPs are filtered correctly. Zero legacy Date objects in the codebase — all java.time.
Week 3 Validation, Error Handling, and Clean API Design ~10 hrs
Mon / Tue
Bean Validation: Add @Valid, @NotNull, @NotBlank, @FutureOrPresent (on scheduledAt), @Size to your request DTOs. When someone POSTs a workout with no activityType or a past scheduledAt, the API should return a clean 400 with a message — never a stack trace.
Wed / Thu
Global Exception Handling: Implement @ControllerAdvice. Define custom exceptions: WorkoutEventNotFoundException, CircleNotFoundException, CircleJoinCodeInvalidException, UnauthorizedAccessException. Map each to the correct HTTP status and a clean JSON error body.
Fri + Weekend
DTOs and MapStruct: Create CreateWorkoutEventRequest, JoinCircleRequest, WorkoutEventResponse, CircleResponse. Use MapStruct for entity↔DTO mapping. Ensure no JPA entity is ever returned directly from a controller endpoint.
DTO vs Entity
Your WorkoutEvent entity is your database row. Your WorkoutEventResponse is what the API returns — it might omit internal fields, add a computed joinCount, or include isExpired. Always separate the persistence model from the API contract.
@ControllerAdvice
One class handles exceptions for all controllers. Throw CircleJoinCodeInvalidException anywhere in a service — it automatically becomes a clean 404 JSON response. Without it, every controller needs its own try/catch. With it, error handling is centralized.
ReadBaeldung — "Bean Validation with Spring Boot" · "Exception Handling for REST with Spring" · "Quick Guide to MapStruct"
Docsmapstruct.org — getting started guide (10 min read)
PracticeHave someone send a badly-formed POST to your API. It should return a clean JSON error body, never a Java stack trace. That's your test.
🏁 Module 1 Complete — Backend Foundation
PaceCircle backend has: layered architecture with DTOs and MapStruct, global error handling via @ControllerAdvice, date-range and upcoming-event queries using java.time, RSVP expiry filtering using Streams, Bean Validation on all input. This is a production-quality Java API.

Weeks 4–6 · Production Patterns
Testing, Profiling & Background Jobs
JUnit 5 Mockito StopWatch Actuator ~30 hrs
▼
Week 4 Testing with JUnit 5 and Mockito ~10 hrs
Mon – Wed
Unit testing with Mockito. Test your service layer by mocking the repository. Learn @Mock, @InjectMocks, @ExtendWith(MockitoExtension.class), when().thenReturn(), verify(mock, times(n)).method(), assertThrows(). The goal: your service tests don't hit a real database.
Thu – Fri
Integration testing with @SpringBootTest. Test the full stack — controller through service through real database (use H2 in-memory). Write a test that POSTs a workout event, then GETs it back, and asserts values match. This is how companies verify APIs don't break between deploys.
Weekend
Test RSVP expiry and circle visibility logic. Edge cases: 0 RSVPs, event time exactly now, user in multiple circles where only one has the event, user who left a circle trying to see old events. Use verify() to assert that service methods only call the repository once — catching accidental N+1 queries.
Unit vs Integration Test
Unit test: tests one class in isolation — mock all dependencies. Runs in milliseconds. Proves: is my logic correct? Integration test: tests multiple layers together — hits a real (test) database. Runs in seconds. Proves: do the layers connect correctly? You need both.
@Mock vs @MockBean
@Mock is pure Mockito — fast, no Spring context. Use in unit tests. @MockBean replaces a Spring bean in the full application context. Use in @SpringBootTest tests. This distinction appears in almost every Spring project.
🏁 Week 4 Milestone
Every service method has at least one unit test. RSVP expiry and circle visibility logic have full edge case coverage. ./mvnw test shows green. You can explain @Mock vs @MockBean.
Week 5 Performance Profiling — StopWatch, Actuator, Logging ~10 hrs
Mon / Tue
Spring StopWatch. Add org.springframework.util.StopWatch profiling to your service methods. Wrap named sections: sw.start("fetch-events"); ... sw.stop(); log.info(sw.prettyPrint()); This outputs a breakdown table of how long each step took. Industry standard for spotting slow service methods before production.
Wed / Thu
Spring Boot Actuator. Add the Actuator dependency. Explore /actuator/health, /actuator/metrics, /actuator/info. Understand what each exposes and why production apps wire these to monitoring systems like Datadog or Grafana. Configure exposure in application.properties.
Fri + Weekend
Structured logging with SLF4J. Replace all System.out.println with proper log statements. Use the correct level for each: log.debug() for dev-only detail, log.info() for normal operations, log.warn() for degraded-but-not-fatal states, log.error() for failures. Never log passwords, JWT tokens, or PII.
org.springframework.util.StopWatch
Spring's built-in timer. Start named tasks, stop them, call prettyPrint() to get a formatted table of task names and durations in ms. Used in real codebases to find which method step is causing latency. Not a tutorial-only tool — this appears in legacy production code.
Actuator / Observability
/actuator/health returns whether your app is alive and DB connection is working. Load balancers ping this to decide if an instance should receive traffic. If it returns DOWN, the instance is pulled from rotation. This is how companies detect outages without manual intervention.
DocsSpring Boot Actuator docs — read "Endpoints" and "Metrics" sections
ReadBaeldung — "A Guide to Spring Boot Actuator" · "Introduction to Spring's StopWatch" · "Logging in Spring Boot"
Week 6 Background Jobs, Scheduling, and Async Processing ~10 hrs
Mon – Wed
@Scheduled tasks. Add a nightly job (2am every day) that marks expired WorkoutJoin records. Add a weekly job (Monday 9am) that soft-deactivates circles with no activity in 90 days. Learn cron expressions: @Scheduled(cron = "0 0 2 * * *"). Use crontab.guru to verify your expressions.
Thu – Fri
@Async methods. When a new member joins a circle, send a welcome email asynchronously — the join endpoint returns 201 immediately, the email sends in the background via CompletableFuture<Void>. Requires @EnableAsync on your main class.
Weekend
Understand the proxy limitation. @Async has the exact same internal-call proxy issue as @Transactional. If you call an @Async method from within the same class, it runs synchronously. Write a test demonstrating this, then fix it by moving the async method to a different bean.
@Scheduled vs @Async
@Scheduled: triggered by time, runs on a timer thread regardless of user actions. @Async: triggered by a user action but doesn't block the HTTP response — runs on a thread pool thread. Both require annotations on your main class. Both share the same internal-call proxy limitation as @Transactional.
202 Accepted
When work happens asynchronously, return 202 instead of 200. 200 means "I finished." 202 means "I received your request, processing is in progress." Knowing this distinction is basic fluency for a backend role.
🏁 Module 2 Complete — Production Patterns
PaceCircle has full test coverage, StopWatch profiling on expensive service methods, Spring Actuator enabled, structured SLF4J logging, a nightly expiry cleanup job, a weekly circle deactivation job, and async email scaffolding that returns 202 immediately. These patterns are what separate a tutorial project from code you'd actually deploy.

Optimistic Updates (React Query)

After a successful POST (new workout logged), call this to force React Query to refetch the calendar data. Without it, the new event won't appear until the next page load. With it, the calendar updates in under a second after the modal closes.

Week 10 Spring Security — JWT Filter Chain + Data Isolation ~10 hrs
Mon / Tue
JWT filter chain. Implement a JwtAuthenticationFilter that validates the token on every request, extracts the userId, and sets SecurityContextHolder. Configure the filter chain: /api/** requires auth, /api/auth/** is public. Disable CSRF — understand why JWT APIs don't need it.
Wed / Thu
Ownership checks in every service method. Read the authenticated userId from SecurityContextHolder. Assert the resource belongs to the requesting user before returning it. For circle-visible events: assert the user is a member of at least one circle the event was posted to. One missing check = a data breach. This is OWASP A01.
Fri + Weekend
Security tests. Write tests proving User A cannot read User B's private events with a valid JWT. Return 401 for missing/invalid tokens. Return 403 for valid token accessing unauthorized resources. These are the two most commonly confused HTTP codes in backend projects — know the exact difference.
401 vs 403
401 Unauthorized: no valid auth presented — "I don't know who you are." 403 Forbidden: valid auth, but no permission — "I know who you are, and you can't do this." Swapping these is a common mistake.
Why JWT APIs skip CSRF
CSRF exploits browsers auto-sending cookies. JWTs must be explicitly added to the Authorization header — browsers never auto-send headers. A malicious site cannot forge a request with your JWT. So CSRF protection is unnecessary for JWT APIs. Disable it: http.csrf(csrf -> csrf.disable()).
🏁 Week 10 Milestone
JWT filter validates every request. Every service method has ownership checks. Security tests confirm cross-user data access is blocked. 401 and 403 are returned correctly. You can explain the CSRF tradeoff.
Week 11 Redis Caching + Rate Limiting ~10 hrs
Mon / Tue
Redis + Spring Cache. Install Redis (brew install redis). Add spring-boot-starter-data-redis + spring-boot-starter-cache. Annotate your circle feed and circle member list service methods with @Cacheable. First call hits PostgreSQL, subsequent calls return from Redis. Profile with StopWatch before and after to see the latency difference.
Wed / Thu
Cache invalidation. When a new workout event is posted to a circle, the feed cache must evict. When a new member joins, the member list cache must evict. Use @CacheEvict on your write methods. Configure TTLs: 1 hour for feed, 24 hours for member lists. Draw a dependency map first — which writes invalidate which caches.
Fri + Weekend
Rate limiting with Bucket4j. Add rate limiting on the feed endpoint — 100 requests per minute per user. Return 429 Too Many Requests when the bucket is empty. Understand the token bucket algorithm. This is a standard protection pattern for any read-heavy endpoint.
@Cacheable + @CacheEvict
@Cacheable: first call stores result in Redis. Subsequent calls return cached result without hitting the DB. @CacheEvict: removes the cached value, forcing the next call to re-execute. The challenge is knowing which writes must evict which caches — this is where bugs live.
TTL vs Explicit Eviction
TTL is the safety net — cache expires after N minutes no matter what. Explicit eviction via @CacheEvict is immediate and precise. Use both: TTL as a fallback for missed evictions, explicit eviction for correctness on write. Without TTL, a bug in your eviction logic means stale data forever.
Week 12 Docker + Cloud Deployment ~10 hrs
Mon – Wed
Multi-stage Dockerfile. Stage 1: maven:3.9-eclipse-temurin-21 runs mvn package. Stage 2: eclipse-temurin:21-jre-alpine (tiny JRE, no Maven, no source) copies only the JAR from Stage 1. Final image ~200MB instead of ~600MB. Write docker-compose.yml that starts API + PostgreSQL + Redis together with one command.
Thu – Weekend
Deploy to Railway + Vercel. Push Spring Boot backend to Railway (Docker-based, managed PostgreSQL plugin). Set all secrets as environment variables: JWT_SECRET, DATABASE_URL, REDIS_URL — never hardcoded. Deploy React frontend to Vercel — set VITE_API_BASE_URL to the Railway URL. PaceCircle is live on the internet.
Multi-Stage Dockerfile
Stage 1 builds the JAR using a full Maven + JDK image. Stage 2 creates the production image with only a slim JRE — no Maven, no source code, no build tools. Smaller image, smaller attack surface. Every company uses this pattern. Knowing it is non-negotiable for a backend role.
Environment Variables
Secrets like JWT_SECRET and DATABASE_URL must never be in source code. Use ${VARIABLE_NAME} in application.properties and set real values on the deployment platform. This is a security baseline — if it's in the repo, it can be leaked.

React Native vs React Web
Same mental model (components, hooks, props, state, TypeScript) but no browser APIs. No DOM, no div, no CSS. Native components compile to real iOS/Android views. The TypeScript and React knowledge from the web version transfers completely. The adjustment is primitives and layout — not architecture.
expo-secure-store
Encrypted key-value storage backed by iOS Keychain and Android Keystore. Use for JWTs, tokens, and any sensitive data. AsyncStorage is unencrypted — never use it for tokens. Security-focused engineers at mobile teams ask about this distinction.

How to Use Your 10+ Hours Per Week

Weekday Structure (~1.5 hrs/day)
▸First 30 min: read one concept — Baeldung article, docs section, or a chapter. Write the key insight in your own words in a running notes doc.
▸Next 60 min: build. Implement what you read into PaceCircle. Code from scratch — no copy/paste from tutorials. If you get stuck, spend 15 min debugging before looking anything up.
Weekend Structure (~4.5 hrs total)
▸Saturday (2–3 hrs): the bigger build — implement the week's milestone feature end to end. This is your main coding session. No interruptions.
▸Sunday (1–2 hrs): write tests for what you built Saturday. Fix anything broken. Document what you learned that week in a short journal entry — this forces retention and becomes your reference project material.
